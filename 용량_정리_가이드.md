# daily-productivity 용량 분석과 컴팩트 운영 가이드

## 1. 캐시는 왜 생기나?

캐시는 쉽게 말하면 **"다음 작업을 더 빨리 하려고 미리 저장해두는 중간 결과물"** 입니다.

예를 들어 설명하면:
- 처음 요리할 때는 재료를 다 꺼내고 손질하느라 오래 걸립니다.
- 그런데 자주 쓰는 재료를 미리 손질해서 냉장고에 넣어두면 다음 요리가 빨라집니다.
- 캐시는 이런 **"미리 손질해 둔 재료"** 같은 개념입니다.

Next.js, TypeScript, React 개발 환경에서는 파일이 많고 변환 과정이 많아서, 매번 처음부터 다시 계산하면 너무 느립니다. 그래서 도구들이 **이전에 계산한 결과를 저장**해둡니다.

## 2. 캐시의 역할은 뭐야?

캐시의 핵심 역할은 하나입니다.

**속도를 빠르게 하는 것**

구체적으로는:
- 개발 서버 실행 속도 개선
- 화면 새로고침 속도 개선
- 빌드 속도 개선
- 타입 검사 속도 개선
- 이전에 처리한 파일은 다시 전부 계산하지 않도록 도움

즉, 캐시는 **용량을 조금 더 쓰는 대신 시간을 절약하는 장치**입니다.

## 3. 캐시에는 어떤 정보가 기록되나?

이 프로젝트 기준으로 보면, 캐시에 저장되는 건 대체로 이런 것들입니다.

### `.next`
Next.js가 만드는 캐시/빌드 결과 폴더입니다.

들어가는 정보 예시:
- 페이지와 컴포넌트를 변환한 결과물
- 브라우저용으로 묶은 JS/CSS 파일
- 서버 실행용 빌드 결과
- 개발 서버가 빠르게 반응하기 위한 중간 산출물
- Turbopack 캐시 파일 (`.sst` 등)
- 어떤 파일을 이미 처리했는지에 대한 내부 메타데이터

쉽게 말하면:
**"소스코드를 바로 실행 가능한 형태로 바꿔 놓은 작업 흔적"** 이 들어 있습니다.

### `tsconfig.tsbuildinfo`
TypeScript가 만드는 증분 빌드 캐시입니다.

들어가는 정보 예시:
- 어떤 TypeScript 파일을 검사했는지
- 이전 검사 시점 정보
- 어떤 파일이 바뀌었는지 판단하기 위한 메타데이터
- 다음 타입 검사 때 재사용할 상태 정보

쉽게 말하면:
**"어디까지 검사했는지 적어 둔 메모장"** 입니다.

## 4. 다시 생기는 거면 또 커지는 거네?

네, 맞습니다.

캐시는 지워도 다시 생깁니다. 왜냐하면 도구가 다시 빨라지기 위해 또 저장하기 때문입니다.

하지만 중요한 포인트는 이겁니다.

- 캐시는 **필요할 때만 다시 생성**됩니다.
- 계속 무한정 커지기만 하는 건 아니고, 보통은 개발 패턴에 따라 어느 정도 범위 안에서 커집니다.
- 특히 `next dev`를 자주 실행하고 파일 수정이 많으면 `.next`가 다시 커질 수 있습니다.
- `node_modules`는 캐시라기보다 **설치된 부품 창고**라서, 지우면 다시 `npm install` 때 원래 비슷한 크기로 생깁니다.

즉:
- `.next` = 작업 속도용 임시 결과물이라 지웠다가 다시 커질 수 있음
- `node_modules` = 프로젝트 실행에 필요한 부품 모음이라 재설치하면 다시 비슷한 크기로 생김

## 5. 앞으로 컴팩트하게 운영하려면?

용량을 항상 아주 작게 유지하고 싶다면, 핵심은 **"항상 필요한 것만 두고, 다시 만들 수 있는 건 자주 비우기"** 입니다.

추천 운영 방식:

1. 평소 보관 시
- `node_modules` 삭제 가능
- `.next` 삭제 권장
- `tsconfig.tsbuildinfo` 삭제 가능

2. 다시 작업 시작할 때
- `npm install`
- `npm run dev`

3. 장기 보관/백업 시
- 소스코드만 남기고 캐시와 설치물은 제외
- 즉, 사실상 아래만 있으면 충분한 경우가 많음
  - `app`
  - `components`
  - `lib`
  - `public`
  - `store`
  - `scripts`
  - `package.json`
  - `package-lock.json`
  - 설정 파일들

## 6. 이 폴더 기준 정리

현재 확인된 대용량 항목:
- `node_modules`: 약 455MB
- `.next`: 약 247MB
- 전체: 약 704MB

즉, 이 폴더가 큰 이유는 거의 전부 아래 때문입니다.
- 설치된 라이브러리
- 개발 캐시

코드 원본 자체는 매우 작습니다.

## 7. 절대 지우지 말 것 / 지워도 안전한 것 / 지우면 다시 설치 필요한 것

| 구분 | 항목 | 역할 | 지워도 되나 | 지우면 어떻게 되나 |
|---|---|---|---|---|
| 절대 지우지 말 것 | `app`, `components`, `lib`, `public`, `store`, `scripts` | 실제 프로젝트 코드와 리소스 | 보통 안 됨 | 프로젝트 내용이 사라지거나 동작이 깨짐 |
| 절대 지우지 말 것 | `package.json` | 어떤 라이브러리를 쓰는지 정의 | 안 됨 | 프로젝트 설치/실행 기준이 사라짐 |
| 절대 지우지 말 것 | `package-lock.json` | 설치 버전 고정 정보 | 가능은 하지만 비추천 | 설치 버전 재현성이 떨어질 수 있음 |
| 절대 지우지 말 것 | `.env.local` | 환경변수, 키, 설정값 | 매우 주의 | API 키나 연결 설정이 사라질 수 있음 |
| 절대 지우지 말 것 | `.git` | Git 이력과 버전관리 정보 | 보통 안 됨 | 저장소 이력과 브랜치 정보가 사라짐 |
| 지워도 안전한 것 | `.next` | Next.js 빌드/개발 캐시 | 됨 | 다음 실행 때 다시 생성됨 |
| 지워도 안전한 것 | `tsconfig.tsbuildinfo` | TypeScript 증분 빌드 캐시 | 됨 | 다음 타입체크/빌드 때 다시 생성됨 |
| 지워도 안전한 것 | 로그 파일들(`npm-debug.log` 등) | 디버깅 기록 | 있으면 삭제 가능 | 필요 시 새 로그가 다시 생성됨 |
| 지우면 다시 설치 필요한 것 | `node_modules` | 설치된 라이브러리 전체 | 됨 | `npm install` 다시 해야 함 |

## 8. 가장 현실적인 컴팩트 운영법

### 자주 작업 중인 프로젝트
- `node_modules`는 유지
- `.next`만 가끔 삭제
- 필요하면 `tsconfig.tsbuildinfo`도 삭제

장점:
- 다시 시작이 빠름
- 용량은 어느 정도 줄일 수 있음

### 당분간 안 건드릴 프로젝트
- `.next` 삭제
- `tsconfig.tsbuildinfo` 삭제
- `node_modules` 삭제

장점:
- 용량을 가장 많이 줄일 수 있음

대신 다시 작업할 때:
- `npm install`
- `npm run dev`

## 9. 한 줄 요약

이 폴더가 큰 이유는 **코드가 많아서가 아니라, 설치된 라이브러리와 개발 캐시 때문**입니다.

가장 안전하게 줄이려면 먼저 `.next`와 `tsconfig.tsbuildinfo`를 지우고,
더 과감하게 줄이려면 `node_modules`도 지우면 됩니다.
